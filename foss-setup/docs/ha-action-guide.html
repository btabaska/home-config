What the plan says (master plan §1): traffic model is “allow Trusted → IoT, block IoT → Trusted, add narrow pinholes.” Your Trusted devices (laptop, phone, mini, HA) should be able to initiate connections to any IoT device; IoT devices can never initiate back. That model is what makes “HA on Trusted controls devices on IoT” work.

What I actually observed today:

  • HA (192.168.10.50, Trusted) can reach the Hue bridge (192.168.20.100, IoT) — the integration is live, lights respond.
  • The mini (192.168.10.2, Trusted) cannot — the gateway actively rejects it (No route to host in 36 ms = a firewall REJECT, not a timeout).

So the real firewall is narrower than the plan’s model — something like “block Trusted→IoT except specific devices,” or an allow rule scoped to HA only. I can’t see UniFi policies (GUI-only, no API creds), so I can’t tell you which. This is not a proposed change to any VLAN membership — mini and HA stay exactly where they are. It’s about which rules exist between the two networks.

Why it matters (concretely):

  • It likely explains your Hue app weirdness. If phones on Trusted WiFi can’t reach VLAN 20, the Hue app can’t connect to the bridge locally — so it shows stale cached info (that 192.168.1.115). Same block would break casting/AirPlay to the TVs once they move to IoT.
  • Verification can’t probe IoT. The daily sweep runs on the mini; it can’t health-check the Hue bridge, ecobee, TVs, etc. if mini→IoT is rejected.
  • Debugging everything IoT becomes guesswork from any machine except HA itself.
