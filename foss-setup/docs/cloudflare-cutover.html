Why your email is safe

Mail delivery follows the records (MX/TXT/DKIM), not the nameservers that serve them. If Cloudflare is already handing out the identical records before you flip the nameservers, every resolver gets the same answer from either side throughout propagation. There is no window where mail has nowhere to go. The one rule: don't flip until the records match.

Two records that must stay grey-cloud (DNS only)

The MX records — Cloudflare forces this automatically; mail can't be proxied. Fine.

The www A record → 192.168.10.2 — that's a private LAN address. If you leave Cloudflare's orange proxy cloud on, Cloudflare will try to reach a private IP from the public internet and the name breaks. Click the cloud to grey. (Your actual services resolve internally via AdGuard, so this record barely matters, but proxying it produces confusing failures.)

The cutover, in order

  1. Add the zone in Cloudflare

    Create a free Cloudflare account, "Add a site" → tabaska.us, choose the Free plan. Cloudflare scans your existing zone and pre-fills what it can find — verify every row against the table above and add anything missing. It usually misses the exact TXT strings; paste those by hand.

  2. Turn off the proxy on www

    Set the www A record to grey-cloud (DNS only). Confirm both MX rows show grey too (forced).

  3. Optional but recommended: add Proton DKIM + DMARC now

    Your zone currently has SPF but no DKIM or DMARC — so mail works, but deliverability isn't as strong as it could be. While you're in the Proton dashboard (Settings → Domain names → tabaska.us), Proton shows three *._domainkey CNAME records and a suggested DMARC TXT. Add them in Cloudflare (all DNS-only). Skip if you want the smallest-possible change; it doesn't block anything.

  4. Copy the two Cloudflare nameservers

    Cloudflare assigns you a pair like xyz.ns.cloudflare.com. Copy both.

  5. Point Squarespace at them

    In Squarespace domain settings, switch from Squarespace/Google nameservers to custom nameservers and paste the Cloudflare pair. This is the actual "flip." Save.

  6. Wait for propagation, then verify

    Usually 15 min–2 h (up to 24 h worst case). Cloudflare's dashboard emails you "tabaska.us is now active." Verify mail records survived with:

    dig +short MX tabaska.us  → should still list the two protonmail hosts.
    dig +short TXT tabaska.us  → should still show the SPF + verification strings.

    Send yourself a test email both directions. If MX resolves, mail is flowing.

  7. Create the scoped API token for Caddy

    Cloudflare → My Profile → API Tokens → Create → Edit zone DNS template. Scope it to Zone : DNS : Edit on tabaska.us only (not all zones). Copy the token once — you can't see it again. This is the last of the three keys on your shopping list.

One thing to know going in

DNS-01 issuance means Let's Encrypt proves domain control by reading a TXT record Caddy writes via the token — your services never get exposed to the internet. That's exactly why this is the right approach for a LAN-only homelab, and why the token only needs DNS-edit permission, nothing else.
