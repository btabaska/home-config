Game hosting on the 24/7 rig

How Minecraft Java and Palworld — and the games that follow — slot into the rig alongside the AI stack, wired into the same dashboard, monitoring, and backups you already run. Written after the 24/7 decision, which simplifies this substantially.

What 24/7 changed

The original spec spent most of its complexity on the rig being asleep: waking it for game night, a friend-wake dashboard (UPSnap), sleep/resume save-hooks, an idle-suspend guard that mustn't kill a live session. All of that is now deleted. The servers simply run. A friend connects any time; the box is already up. What remains is the actual game-hosting stack — smaller and more reliable than the on-demand version would have been.

Does it fit alongside the AI stack?

Comfortably. The AI stack idles at ~3 GB resident (it only balloons during inference, which is bursty and rarely concurrent with game sessions). Minecraft Paper wants ~10 GB, a Palworld dedicated server ~16–24 GB under a full lobby. Even with both games full and a model loaded, you stay inside 62 GB with real headroom.

The GPU is untouched by either game server — dedicated servers are CPU/RAM/network workloads, so the 3090 Ti stays fully available for AI and for Apollo game streaming (Apollo is a maintained Sunshine fork — same NVENC path, drop-in for Moonlight). The 300 W power cap and the new suspend-masking mean the box behaves predictably under mixed load.

The stack

One new Compose project at /opt/stacks/games/ on the rig, following the same pinned-image, one-service-per-game pattern as the rest of the fleet. Nothing bespoke — both images are the community-standard, actively-maintained choices.

rig · /opt/stacks/games/ (Compose)
Pinned images, restart: unless-stopped, per-game data volumes under the restic backup path.
minecraft · itzg/minecraft-server (Paper)
10 GB Aikar-tuned heap · Geyser + Floodgate so your Switch 2 / Bedrock devices join the same world · mc-backup sidecar for world snapshots · container auto-pause (freezes when empty, wakes on connect — saves CPU, not power).
palworld · thijsvanloef/palworld-server-docker
Performance flags on · scheduled nightly restart (works around the known Palworld memory leak) · RCON for graceful saves · password-gated (Palworld has no real auth — RCON/REST never exposed publicly).
managed in Dockge · surfaced on home.tabaska.us
The games stack shows up in the Dockge UI you already use (start/stop/logs, no new panel to learn) and gets Homepage tiles with live status.
On control panels (Pelican / Pterodactyl)

I recommend not running a dedicated game panel yet. Pelican is still pre-1.0, and a panel's main value — provisioning servers on demand — duplicates what Compose + Dockge already give you for two servers. It adds a database, a daemon, and its own auth surface for little gain at this scale. Revisit when you're running 5+ servers or want friends to self-provision. For now, Dockge is the panel.

Security line I won't cross without you

Palworld has no authentication beyond a server password, and its RCON/REST admin ports are unauthenticated-adjacent. Those never get exposed publicly — friend access is Tailscale (private mesh) or playit (game port only). Exposing a game port to the open internet is a deliberate, separate decision, not something I'll do as part of the build.

Proposed rollout

  1. Substrate

    Create /opt/stacks/games/, the Compose scaffold, backup wiring, and monitoring hooks. No game running yet — just the frame, committed and verified. (fully autonomous)

  2. Minecraft first

    Paper + Geyser + mc-backup. I bring it up, smoke-test a local connect and a backup/restore, add the Uptime monitor + Homepage tile. You then connect from your Switch to confirm cross-play.

  3. Palworld second

    Dedicated server with the leak-restart schedule and RCON saves, monitored the same way. Validated the same way.

  4. Friends + future games

    Tailscale sharing / playit for external players (your steps), then the pattern repeats for each new game — same Compose shape, same monitoring, same backups.
