## About

Vaultwarden is a self-hosted, Bitwarden-compatible password-manager server (`vaultwarden/server:1.36.0`, shipping web vault v2026.4.1) running on `mini` from `foss-setup/configs/docker-stack/stacks/vaultwarden/compose.yaml` and reachable at https://vault.tabaska.us. It publishes no host port; Caddy fronts it over the shared external `edge` network (`reverse_proxy vaultwarden:80`, websockets handled automatically), so its `DOMAIN` env var must exactly match the public URL for links, invites, and WebAuthn to work. State is a small SQLite DB plus attachments under `./data` (bind-mounted `./data:/data`), intentionally INCLUDED in restic backups (see `/etc/restic/excludes.txt`). The `/admin` panel is gated by `ADMIN_TOKEN` (recorded in the operator vault, `.handoff-secrets.yaml`), and `SIGNUPS_ALLOWED` should be flipped to false once the operator has registered. Task foss-01 is the in-progress Bitwarden to Vaultwarden data cutover.

## Troubleshooting

- **Login links, invites, or WebAuthn/passkeys fail or point to the wrong host** — Ensure `DOMAIN` in `/opt/stacks/vaultwarden/.env` is set to the exact public URL (`https://vault.tabaska.us`) that clients use; correct it then `ssh mini 'cd /opt/stacks/vaultwarden && docker compose up -d'`.
- **Anyone can self-register a new account** — After the operator has registered, set `SIGNUPS_ALLOWED=false` in `.env` and `docker compose up -d`; the compose default is already false but `.env.example` ships it true for initial setup.
- **/admin panel unreachable or asks for a token that doesn't work** — `ADMIN_TOKEN` must be present in `.env` (generate with `openssl rand -base64 48`); the live token is also in the operator vault `.handoff-secrets.yaml`.
- **Container shows unhealthy after a restart** — Healthcheck runs the image's `/healthcheck.sh` (curls /alive) with a 40s start_period; check `ssh mini 'cd /opt/stacks/vaultwarden && docker compose logs --tail 50'` and confirm `Rocket has launched from http://0.0.0.0:80`.
