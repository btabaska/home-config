## About

Miniflux is a minimalist, single-binary Go RSS/Atom reader backed by PostgreSQL, running on `mini` (192.168.10.2) from `foss-setup/configs/docker-stack/stacks/miniflux/compose.yaml` and served at https://rss.tabaska.us. It is a two-container stack: `miniflux/miniflux:2.3.1` (host port `8082`→container `8080`) and a private `postgres:17-alpine` db, connected via a bridge `miniflux` network for the DB plus the shared external `edge` network so Caddy fronts it (the container also stays directly reachable on LAN port 8082). `DATABASE_URL` uses `sslmode=disable` over the internal network and `RUN_MIGRATIONS=1` auto-migrates on boot. The single user is `btabaska`; the one-time `CREATE_ADMIN` bootstrap env and its leftover `admin` account were removed 2026-07-16 (fix-33, quality-gate L23), and an API key for `btabaska` lives in the secrets vault under `miniflux.api_key` (send it in an `X-Auth-Token` header). `POLLING_PARSING_ERROR_LIMIT=0` is set deliberately so feeds are never auto-excluded after consecutive fetch errors — see the frozen-feeds entry below. Postgres is deliberately pinned to `17-alpine` because pg18 moved the data path — the `./db:/var/lib/postgresql/data` mount would break on an unadjusted major bump.

## Troubleshooting

- **No new articles for days; every feed shows a "network error" from the same moment; `docker logs miniflux` shows hourly `Created a batch of feeds ... jobs_count=0`; UI and healthcheck stay green** — The feeds were auto-excluded from polling: Miniflux permanently skips any feed whose `parsing_error_count` reaches `POLLING_PARSING_ERROR_LIMIT` (default 3), so a transient outage lasting a few poll cycles — like the 14-hour docker-DNS outage of 2026-07-08/09 (quality-gate H7) — silently freezes the entire reader while liveness checks stay green. Recover by resetting the counters and forcing a refresh: `ssh mini docker exec miniflux_db psql -U miniflux -d miniflux -c "UPDATE feeds SET parsing_error_count=0, parsing_error_msg=''"`, then `PUT /v1/feeds/refresh` with the vault API key (`curl -X PUT -H "X-Auth-Token: $KEY" http://192.168.10.2:8082/v1/feeds/refresh`). The stack now sets `POLLING_PARSING_ERROR_LIMIT=0` (never exclude — dead feeds just keep retrying hourly and stay visible in the UI), and the `mini-miniflux-feeds-fresh` / `mini-miniflux-articles-flowing` / `mini-container-dns` verification checks alert on scheduler stalls, zero-ingest, and the root-cause container-DNS failure respectively.
- **DATABASE_URL fails to parse / miniflux can't reach the DB after setting a password** — MINIFLUX_DB_PASSWORD must be URL-safe — a `/` or `+` in the password breaks the `postgres://...@db/miniflux` connection string. Regenerate with `openssl rand -hex 24` (hex-only) and update `.env` on the host.
- **Postgres won't start after bumping the `db` image to Postgres 18** — pg18 changed PGDATA from `/var/lib/postgresql/data` to `/var/lib/postgresql`. The stack pins `postgres:17-alpine` for this reason; if moving to 18, update the volume mount in `compose.yaml` accordingly (and dump/restore across a major version — data dirs are not forward-compatible).
