## About

Pinchflat is a "Sonarr for YouTube" channel/playlist archiver — a single self-contained Elixir/Oban app built on `yt-dlp` — running on `mini` from `foss-setup/configs/docker-stack/stacks/pinchflat/compose.yaml` (container `pinchflat`, port `8945`, fronted by Caddy at https://pinchflat.tabaska.us on the external `edge` network). It subscribes to channels/playlists, downloads new uploads on a schedule, applies SponsorBlock, and writes clean NFO metadata + thumbnails into `${PINCHFLAT_DOWNLOADS}` (`/mnt/nas-youtube/pinchflat` → the NAS "YouTube" Plex library path) so Plex indexes the archive as a show. The image is NOT stock upstream: it is a locally-built `pinchflat-bgutil:local` (`Dockerfile` `FROM ghcr.io/kieraneglin/pinchflat:latest`) that layers the nightly `yt-dlp` zipapp, a baked `/etc/yt-dlp.conf` (alternate player clients against the 403/JS-challenge, plus `youtubepot-bgutilhttp:base_url=http://bgutil-pot:4416`), and — since fix-37 (M14, 2026-07-18) — the bgutil PO-token plugin zip in `/etc/yt-dlp/plugins/`, pinned to the same version as the bgutil-pot server (1.3.1, bump together). Before that, the image's name promised bgutil but never installed the plugin, so the POT provider sat idle while 7 videos stranded on "Sign in to confirm you're not a bot" (2026-07-14); with the plugin wired, every download fetches a PO token (verified live: provider generated its first POT since 07-08 on the fix day). Refresh with `docker compose build --pull && docker compose up -d`; the `pinchflat-pot-provider` check fails if a rebuild loses the plugin/conf or the provider goes unreachable.

## Troubleshooting

- **A large bulk ingest from Pinchflat makes ALL Plex clients return 503 while the process shows "running" and drains glacially.** — Bulk ingest floods Plex's none-agent metadata analysis and saturates the analysis worker pool. Diagnose via the Plex Media Server log (live analysis count / 408 trend); there is no fast fix — let it drain, or throttle Pinchflat's download concurrency and stage large channel back-catalogs in smaller batches rather than all at once.
- **Plex shows 0 items in the YouTube library even though Pinchflat wrote the files.** — Files written to the NAS CIFS/DSM share by Pinchflat lack the PlexMediaServer ACL ACE, so Plex cannot read them. Apply the synoacltool fix + enable inheritance on the share (same class of fix as the NAS Plex share ACL issue). Confirm inheritance so newly-downloaded files inherit the ACE automatically.
- **Downloads fail with HTTP 403 / "Sign in to confirm you're not a bot" / n-signature errors.** — Three escalating layers, all baked into the image since fix-37: (1) stale yt-dlp — rebuild for the latest nightly: ssh mini 'cd /opt/stacks/pinchflat && docker compose build --pull && docker compose up -d'; (2) PO tokens — verify the bgutil plugin is live with the pinchflat-pot-provider check or docker exec pinchflat yt-dlp -v --simulate <url> (look for "bgutil:http-x.y.z (external)" WITHOUT ", unavailable"; plugin and bgutil-pot server versions must match); (3) hard LOGIN_REQUIRED — if a video bot-checks even with a POT attached (confirmed for the 7 items stranded 2026-07-14; ids 409/702/895/915/939/1008/1333 accepted in the pinchflat-stuck-media check), the egress IP is hard-flagged for that video and only sign-in cookies would recover it — a deliberate non-goal (account-flagging risk). New items in that state page via pinchflat-stuck-media.
