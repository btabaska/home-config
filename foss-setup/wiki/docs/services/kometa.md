## About

Kometa (formerly Plex-Meta-Manager) is a scheduled batch job that builds collections, overlays, and playlists across the Plex libraries, pinned at `kometateam/kometa:v2.3.1` and running on `mini` from `foss-setup/configs/docker-stack/stacks/kometa/compose.yaml`. It has no web UI, no port, and joins no shared network — the container stays up (`restart: unless-stopped`) and runs itself daily at `KOMETA_TIME` (05:00), or one-shot via `docker exec -e KOMETA_RUN=True kometa python3 kometa.py` — NOTE the env var is mandatory: Kometa gives environment variables precedence over CLI flags, and the compose env pins `KOMETA_RUN=False`, so the widely-documented `docker compose run --rm kometa --run` silently ignores `--run` and sits waiting for the 05:00 schedule (fix-37 burned 15 minutes on this). All behaviour is driven by `./config/config.yml` (bind-mounted `./config:/config`), which points Plex at the NAS server `http://192.168.10.4:32400` and manages the `Movies` and `TV Shows` libraries plus the MDBList-backed default playlists. fix-37 (M16, 2026-07-18) cut the config to the connections actually in use: plex + tmdb + mdblist (key required — MDBList made keys mandatory and the daily run 401ed for weeks, silently never updating 2 playlists), deleted ~12 empty template integration blocks that each logged a Config Error every run, and dropped the dead `Anime` (no such Plex library) and `Music` (missing Music.yml) library blocks. The `kometa-run-clean` verification check parses the last run's log and fails on ANY buried builder/config error — Kometa exits 0 even when builders inside the run failed, so exit-code monitoring is worthless here. It sits downstream of the *arr/Plex pipeline as pure cosmetic enrichment — it never ingests media, only decorates what Plex already has.

## Troubleshooting

- **Daily run finishes but the Error Summary shows "MDBList Error: 401 - Authentication required" and MDBList-backed playlists silently stop updating.** — MDBList requires an API key for every list fetch. Set mdblist.apikey in /opt/stacks/kometa/config/config.yml (key lives in the secrets vault at mdblist.api_key; free from mdblist.com -> Preferences -> API), then verify with the kometa-run-clean check or a one-shot run: docker exec -e KOMETA_RUN=True kometa python3 kometa.py (the env var is required — see About).
- **Error Summary shows "sub-attribute apikey/token is blank" Config Errors for an integration (notifiarr, omdb, tautulli, radarr, sonarr, trakt, mal, gotify, ntfy...).** — An integration block exists in config.yml without a credential — an empty block is a latent error on every run, not a harmless placeholder. Delete the block entirely (fix-37 removed all of them); re-add it from the upstream template only on the day you actually connect that service, credential included. The kometa-run-clean check pages if any run logs a Config Error again.
- **Log error: "Plex Library 'X' not found. Options: [...]" or "File does not exist /config/Music.yml".** — A libraries: block in config.yml names a Plex library that does not exist, or references a collection file that was never created. Fix the name to a real library (current Plex libraries: Movies, TV Shows, Music, YouTube) or delete the block — fix-37 removed the dead Anime and Music blocks this way.
- **Kometa v2.3.1.4 reports "Newest Version: 2.4.4" — image is behind upstream.** — Version is pinned in compose.yaml on purpose. To update, bump image: kometateam/kometa:<tag> in compose.yaml, then ssh mini 'cd /opt/stacks/kometa && docker compose pull && docker compose up -d'; review upstream changelog since overlay/collection schema can shift between minors.
