## About

A fully LAN-local AI journaling stack on `mini` at `/opt/stacks/journaling/` (mirrored to `foss-setup/configs/docker-stack/stacks/journaling/`), stood up by task `journal-01`. Three digest-pinned containers: **Memos** (`neosmemo/memos:0.29.1`, host `:5230`, fronted by Caddy at https://memos.tabaska.us) is the journal front-end; **n8n** (`n8nio/n8n:2.32.2`, host `:5678`, https://n8n.tabaska.us) is the automation layer that will react to each new entry; and **faster-whisper** via Speaches (`ghcr.io/speaches-ai/speaches:0.8.3-cpu`, host `:8010`) is the internal-only server-side dictation endpoint — deliberately given NO Caddy vhost. Inference is NOT local to this stack: n8n calls the rig's llama-swap at `http://192.168.10.12:9292/v1/chat/completions` with model `dolphin-venice-24b` (temp ~0.35, an uncensored-but-steerable coach), because the mini has no GPU and the capable models live behind llama-swap, not the small-model Ollama `:11434`. faster-whisper runs on CPU for the same reason (the fleet's only GPU, the rig 3090 Ti, is pinned to LLM + Immich ML). All state (Memos sqlite/uploads, n8n workflows/encryption-key) is bind-mounted under the stack dir so restic backs it up. Ports 5230/5678/8010 were chosen because `:8000`/`:8080` are already taken on the mini. Whole stack is LAN/tailnet only, no public exposure and no cloud/telemetry (n8n diagnostics + version checks disabled); the single outbound call is faster-whisper's one-time model pull from HuggingFace. `journal-01` stood up the scaffold; `journal-02` wired the event pipe: the host Memos account (`btabaska`, creds at vault `journaling.memos.*`), a never-expiring Personal Access Token (`memos_pat_…`, vault `journaling.memos.api_token`, also in the live `.env` as `MEMOS_API_TOKEN` for n8n), and a Memos webhook on **every** memo event → the n8n container URL `http://n8n:5678/webhook/journal`. Memos runs with `--allow-private-webhooks` (compose `command`) so it may POST to that private container name; public user-registration was closed (`instance/settings/GENERAL.disallowUserRegistration=true`). A live memo was confirmed to deliver a `memos.memo.created` event to n8n. The validated 0.29 payload/API shapes are recorded in `docs/journaling-stack-plan.md`. `journal-03` built the **`journal-analyze`** n8n workflow (in `n8n/journal-analyze.workflow.json`) that now owns `POST /webhook/journal`, superseding the `journal-webhook-probe` (kept but deactivated). Its pipeline: webhook → a loop-safe **Guard** (Code node) that proceeds only on `memos.memo.created` events carrying a `#journal` tag and never on its own output — it drops `memos.memo.comment.created` outright and skips any memo whose content starts with the `🧭 **Reflection**` sentinel or carries `#analyzed` (essential, because writing a comment fires **two** events both carrying the comment memo) → an **HTTP** call to the rig coach model (120 s timeout, 3 retries, continue-on-error) → a **tolerant JSON parse/repair** (strips ``` fences, grabs the first `{…}`, and closes a truncated object) so a chatty small-model reply never breaks the loop → **write-back as a Memos COMMENT** (`POST /api/v1/memos/{name}/comments`, never a body edit — that would re-fire the webhook) formatted `🧭 **Reflection** · mood N/10 · sentiment` + emotions/themes + one gentle nudge. A VRAM-evicted/OOM'd model just yields no comment (best-effort; the entry is always saved). The workflow reads `MEMOS_API_TOKEN`/`LLM_MODEL`/`LLM_BASE_URL`/`LLM_TEMP` from the n8n container env via `$env`, so swapping the coach model (e.g. to `deckard-heretic`) is a one-line `.env` change + `docker compose up -d n8n`. Validated end-to-end: a `#journal` and a `#gamelog` entry each produced exactly one reflection comment with the two comment-triggered events skipped by the guard. `templates/daily.md` and `templates/gamelog.md` (both carry `#journal`) ship in the stack dir. `journal-04` added the optional **server-side dictation branch**: on `memos.memo.created` the Guard reads `memo.attachments[]` (which the webhook carries inline) and, if the first attachment is `audio/*`, an **IF** node routes to a fetch→transcribe→merge sub-path — otherwise it goes straight to the LLM, so the primary client-side text path is completely untouched. The audio path GETs the attachment bytes from Memos (`/file/{attachment}/{filename}`, Bearer PAT), POSTs them to faster-whisper by container name (`WHISPER_BASE_URL` default `http://faster-whisper:8000/v1/audio/transcriptions` — port 8000 is the *container* port, host-published as 8010; multipart, model `WHISPER_MODEL` default `Systran/faster-whisper-small` on CPU), and merges the returned transcript into the entry as `…\n\n[Voice note transcript]\n<text>` **before** the coach LLM sees it. Both new HTTP nodes are continue-on-error and the merge falls back to the text-only messages, so whisper being down never blocks or regresses a normal entry (best-effort, same posture as the coach model). The Guard also relaxes its empty-body drop when an audio attachment is present, so a voice-only entry (just `#journal` + audio) still gets analyzed. Verified end-to-end: a `#journal` memo carrying a spoken `.wav` was transcribed and its reflection referenced content that appeared **only** in the audio, while a text-only memo skipped the whole branch (the Fetch/Transcribe/Merge nodes never executed). A daily consumer check `journaling-whisper-transcribes` transcribes a bundled 1 s probe to prove the model still loads (not just `/health`). `journal-05` added the **Open WebUI coaching front-end** on the rig (`https://ai.tabaska.us`), a second way to write entries: a **"Journaling Coach"** Workspace model preset (base `dolphin-venice` → llama-swap `dolphin-venice-24b`, temp 0.35) with a gentle-nudge conversational system prompt (free-form — distinct from the analyzer's strict-JSON prompt), plus a **"Save to Journal"** Action function (`save_to_journal_function.py`) that adds a button under each message which POSTs the whole conversation to Memos as a `#journal` memo (`POST /api/v1/memos`, `{"content","visibility"}`, Bearer PAT). Saving this way **intentionally re-enters** the `journal-analyze` loop, so the saved conversation gets its own reflection comment — exactly one, because the loop guard's `#journal`-tag gate + `🧭 **Reflection**` sentinel still hold (the transcript header uses 🪞, not the 🧭 sentinel). Both artifacts live ONLY in the rig Open WebUI database — no compose file owns them — so they are installed via the OWUI admin REST API (`POST /api/v1/functions/create` + `/toggle` + `/valves/update` for the function; `POST /api/v1/models/create` for the preset) reusing the admin key at vault `ai_stack.openwebui_rag_sync_api_key`; the Memos base URL + token are function **Valves** (`MEMOS_BASE_URL` = the mini over the LAN `http://192.168.10.2:5230`, `MEMOS_TOKEN` = vault `journaling.memos.api_token`), never hardcoded. Source of truth for both is `openwebui/{save_to_journal_function.py,journaling-coach-preset.md}` in the stack mirror. Because they are DB-only, two `journal-05` drift-guard checks (`journaling-owui-save-fn-installed`, `journaling-owui-coach-preset`) probe the rig OWUI admin API from the mini to confirm the button + preset are still installed and active — an OWUI redeploy/volume-wipe would silently erase them while every container stays green. Verified end-to-end: a real coaching turn against the preset (`dolphin-venice-24b`, GPU had headroom so this ran on the production model — the first `journal-*` E2E not on the fast-3b stand-in) → **Save to Journal** → the transcript landed in Memos as a `#journal` memo and received exactly one `🧭 **Reflection**` comment (mood 5/10, no re-trigger). `journal-06` closed the stack out. It ships a stack `README.md` (first-run bring-up, the Memos-PAT + webhook walkthrough, the n8n 2.x publish/restart trap, end-to-end testing, backup/export, and full troubleshooting) and the consumer-end monitoring the earlier tasks deferred. The deep end-to-end check **`journaling-loop-e2e`** (helper `verification/bin/journaling-e2e.py`, `tier: daily`) posts a marked `#journal` test memo and asserts the loop writes back **exactly one** reflection comment, then deletes the probe — proving webhook → Guard → rig coach → comment write-back all work *and* the loop guard holds, not merely that containers are up (mandate 1). Because the reflection is best-effort, the probe first checks the coach model can actually load and cleanly **SKIPs** (posting nothing) when the 3090 Ti is too tight to hold the 24 B model — an expected, separately-monitored condition, not a loop failure — so a real FAIL (`E2E_NO_COMMENT`/`E2E_LOOP_MULTICOMMENT`) means the loop itself is broken while the model was available. Both paths were validated at build: a full `E2E_OK` on the `fast-3b` stand-in (the 24 B coach was VRAM-contended by Immich ML at the time — all 24 B models OOM'd with 19 GB free) and a correct `E2E_SKIP_COACH_UNAVAILABLE` against the production `dolphin-venice-24b`. Two Uptime-Kuma liveness tiles (`Mini Memos`, `Mini n8n`, health endpoints, seeded by `scripts/uptime-kuma/seed-monitors.sh`, alerting to ntfy) round out coverage → **10** journaling verification checks green plus the two Kuma tiles up. Backup/export: the whole stack dir (Memos sqlite + uploads, n8n workflows **and** encryption key, Whisper cache) is already in the nightly restic → Backblaze B2 snapshot (confirmed present in `restic ls latest`); a portable JSON **export** was added and tested once — `scripts/export-journal.sh` dumps every entry *and its reflection comments* via the Memos API (the list API returns top-level entries only, so it fetches each entry's `/comments`). A scoped `.gitignore` keeps the live bind-mount state + the n8n encryption key + exports out of the repo (source-only mirror). Coverage unchanged — no new container; all three still in `verification/coverage/mini.containers`.

## Troubleshooting

- **Journal entries save fine but no reflection comment ever appears.** — Reflections depend on the rig, which is a separate host. Confirm the coach model is reachable and registered: `curl -s http://192.168.10.12:9292/v1/models | grep dolphin-venice-24b` (the `journaling-coach-model-reachable` verification check does this). llama-swap loads the ~17 GB model on first call and a rig "gaming force-unload" hook can evict it (Immich ML also holds ~12 GB of the 24 GB card), so analysis is best-effort — the entry is always saved regardless, and the `journal-analyze` workflow retries 3× then posts nothing rather than erroring. Also confirm the workflow is armed (the `journaling-analyze-armed` check: `curl -s http://localhost:5678/webhook/journal` must reply `…not registered for GET requests`, which n8n returns only when a POST webhook actually owns that path).
- **Memos entries are created but n8n never receives a webhook (no executions).** — Check the wiring the `journaling-memos-webhook-wired` verification check guards: the Memos webhook must still point at `http://n8n:5678/webhook/journal` — `curl -s -H "Authorization:Bearer $MEMOS_API_TOKEN" http://localhost:5230/api/v1/users/btabaska/webhooks` (token in `/etc/verification/env`). Memos silently drops private-IP webhook targets unless it runs with `--allow-private-webhooks` (compose `command:` on the `memos` service) — verify with `docker inspect memos --format '{{json .Config.Cmd}}'`. On the n8n side the receiving workflow must be **active AND published**: n8n 2.x needs a *published version* (`docker exec n8n n8n publish:workflow --id=<id>`) — a bare `active=1` flag alone registers nothing (boot logs "Processed 0 published workflows") — then a **restart** to register. Two traps: (1) the CLI `unpublish`/re-`import` leaves a stale `webhook_entity` row that makes the next workflow fail activation with "URL path … already taken" — clear it with n8n stopped (`DELETE FROM webhook_entity WHERE workflowId='<old-id>'`, then start); (2) `import:workflow` deactivates the published version, so re-`publish` after every import. The `journaling-analyze-armed` check catches a silently-unregistered webhook. Note n8n uses SQLite WAL — when reading executions off disk, copy `database.sqlite` **and** its `-wal`/`-shm` files or recent rows are invisible.
- **memos.tabaska.us or n8n.tabaska.us returns a TLS/connection error right after a deploy.** — Both are new Caddy vhosts on the mini `edge` network. Confirm Caddy reloaded after the Caddyfile edit (`docker exec caddy caddy reload --config /etc/caddy/Caddyfile`) and that both containers are on the `edge` network (`docker inspect memos --format '{{json .NetworkSettings.Networks}}'`). A cold first request can 000 while the cert warms; retry once.
- **One of the three containers is unhealthy or the healthcheck.sh reports DEGRADED.** — Run the stack healthcheck from the mini: `ssh mini 'cd /opt/stacks/journaling && ./scripts/healthcheck.sh'` — it probes Memos `:5230/healthz`, n8n `:5678/healthz`, faster-whisper `:8010/health` and rig `:9292/v1/models`. For a stuck container, `cd /opt/stacks/journaling && docker compose logs --tail 50 <service>` then `docker compose up -d`. n8n and whisper bind-mount dirs must stay owned by uid 1000.
- **A voice memo (audio attachment) is saved and analyzed, but the reflection ignores the spoken words — only the typed text was considered, or a voice-only entry got no reflection.** — The `journal-04` transcription branch depends on faster-whisper actually transcribing, which `journaling-whisper-ready` (a bare `/health`) does NOT prove — run `journaling-whisper-transcribes` (the daily consumer check) or POST the probe by hand: `curl -s -m 90 -X POST http://localhost:8010/v1/audio/transcriptions -F file=@/opt/verification/assets/whisper-probe.wav -F model=Systran/faster-whisper-small -F response_format=json`. Common causes: (1) the model isn't cached — `ls /opt/stacks/journaling/whisper-cache/hub/models--Systran--faster-whisper-small` should exist; the container lazy-downloads it on first call and needs DNS (compose sets `dns: 192.168.10.4`) for that one-time pull. (2) n8n can't reach whisper by name — from n8n the URL is the **container** port `http://faster-whisper:8000/v1` (host `:8010` is a publish mapping, NOT what n8n uses); test `docker exec n8n wget -qO- http://faster-whisper:8000/health`. (3) The IF node didn't route to the audio branch — the memo's attachment must be `audio/*`; inspect the execution (copy `database.sqlite` + `-wal`/`-shm`) and confirm `Fetch audio`/`Transcribe`/`Merge` ran. Whisper failing never breaks a normal entry: both HTTP nodes are continue-on-error and the merge falls back to text-only, so the entry is still analyzed on its typed content.
- **In Open WebUI the "Journaling Coach" model or the "Save to Journal" button is gone, or Save to Journal errors / saves nothing.** — The coach preset + save action live ONLY in the rig Open WebUI SQLite DB (`docker_open_webui_data`), not in any compose file, so an OWUI redeploy or volume wipe erases them — the `journaling-owui-save-fn-installed` and `journaling-owui-coach-preset` checks catch this. Reinstall from the stack mirror `openwebui/` via the OWUI admin API (admin key = vault `ai_stack.openwebui_rag_sync_api_key`, already in `/etc/verification/env` as `OWUI_API_KEY`): `POST /api/v1/functions/create` (id `save_to_journal`, `content` = the .py source; type `action` is inferred from `class Action`, do not pass a `type`), then `POST /api/v1/functions/id/save_to_journal/toggle` to activate, then `POST /api/v1/functions/id/save_to_journal/valves/update` with `MEMOS_BASE_URL`/`MEMOS_TOKEN`; and `POST /api/v1/models/create` (id `journaling-coach`, `base_model_id` `dolphin-venice`, `params.system` = the coaching prompt, `params.temperature` 0.35). If a just-created preset returns `{"detail":"Model not found"}` on the first `/api/chat/completions` call, OWUI's completion-path model cache is stale — it self-refreshes within ~a minute (or restart `open-webui`); the base model works immediately. If Save to Journal reports a failure, check the `MEMOS_TOKEN` valve is set (the function refuses to post with an empty token) and that the rig can reach the mini Memos (`ssh rig curl -s -o /dev/null -w '%{http_code}' http://192.168.10.2:5230/healthz`). If the coach itself 500s mid-chat, the 24 B model was evicted for VRAM — free the GPU or temporarily point the preset's base model at `chat`/`fast`.
- **You need to restore the journal, migrate it to another Memos, or confirm the backup actually captures it.** — The complete, automatic **backup** is restic → Backblaze B2 (nightly `restic-backup.timer` on the mini; `BACKUP_PATHS` includes `/opt/stacks`). Confirm the stack is in the latest snapshot: `sudo bash -c '. /etc/restic/env; restic ls latest /opt/stacks/journaling'` — it should list `memos/` (sqlite + uploads), `n8n/` (workflows + encryption key), and `whisper-cache/`. Restore one path: `sudo bash -c '. /etc/restic/env; restic restore latest --target /restore --include /opt/stacks/journaling'`. The n8n **encryption key** (`n8n/config`) rides in the backup but is gitignored out of the repo (source-only mirror) — without it n8n cannot decrypt saved credentials. For a portable, human/other-Memos-readable copy use the **export** instead of a DB dump: `cd /opt/stacks/journaling && ./scripts/export-journal.sh` writes `journal-export-YYYY-MM-DD.json` with every entry **and** its reflection comments (the Memos list API returns entries only, so the script fetches each entry's `/comments`).
