## About

Scrutiny is the fleet's per-disk **S.M.A.R.T. health** monitor — the one hardware blind spot Beszel (host/container metrics) and Uptime-Kuma (service liveness) do not cover, so a failing or aging drive surfaces before it dies (glue-10). This NAS container is the **omnibus** (`ghcr.io/analogj/scrutiny` master-omnibus, digest-pinned) from `foss-setup/configs/nas/scrutiny/docker-compose.yml`: web UI + bundled InfluxDB + the local NAS collector in one image, published on host `:8080` and fronted by Caddy at https://scrutiny.tabaska.us (no auth — LAN/Tailscale only). It is the hub for the whole fleet: the `mini` and `rig` hosts run collector-only `scrutiny-collector` agents that POST their SMART data here. Synology exposes drives as `/dev/sataN` and `smartctl --scan` does NOT enumerate them, so each bay is declared explicitly with `type: sat` in `config/collector.yaml` (verified: `smartctl -d sat -i /dev/sataN` returns full ATA SMART on all three NAS bays). Collectors sweep daily at 06:00 (`COLLECTOR_CRON_SCHEDULE`); the consumer-end check `sys-disk-smart-health` asserts all 7 fleet disks are present with 0 failed status. The Homepage Infrastructure tile uses the native `scrutiny` widget (passed/failed/unknown) pointed at the NAS IP.

## Troubleshooting

- **Dashboard is empty, or one host's disks are missing right after deploy.** — Collectors only run on the daily 06:00 cron, so a freshly-started collector has not posted yet. Force one immediately: on the hub run `sudo /usr/local/bin/docker exec scrutiny /opt/scrutiny/bin/scrutiny-collector-metrics run` (NAS disks); on a spoke run `docker exec scrutiny-collector /opt/scrutiny/bin/scrutiny-collector-metrics run` (mini/rig). Then re-check `curl -s http://192.168.10.4:8080/api/summary`.
- **NAS drives are absent or show no SMART attributes even though the collector ran.** — Synology `/dev/sataN` devices are not found by `smartctl --scan`, so they must be listed explicitly in `/volume1/docker/scrutiny/config/collector.yaml` with `type: 'sat'` under `devices:`. Confirm a device reads with `sudo /usr/local/bin/docker exec scrutiny smartctl -d sat -i /dev/sata1`; if you add a 4th bay, add its `/dev/sataN` to both `devices:` in collector.yaml and the `devices:` list in docker-compose.yml, then `docker compose up -d`.
- **The sys-disk-smart-health check reports smart_FAIL or the drive count dropped below 7.** — Open https://scrutiny.tabaska.us and find the drive whose status is not "Passed". A real SMART fail (device_status non-zero) means a degrading/failing disk — plan a replacement and check the drive's attributes (reallocated sectors, pending sectors, wear). If instead the count dropped, a collector is down: verify the `scrutiny-collector` container is running on the affected host (`docker ps`) and that it can reach the hub at `192.168.10.4:8080`.
