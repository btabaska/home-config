## About

Beszel is an ultra-lightweight server monitoring hub + agent stack running on `mini` from `foss-setup/configs/docker-stack/stacks/beszel/compose.yaml`. The `henrygd/beszel:0.18.7` hub serves the dashboard/REST API on `8090:8090` (fronted by Caddy at https://status.tabaska.us via the external `edge` network), while the `henrygd/beszel-agent:0.18.7` container runs `network_mode: host` and reports this box's CPU/memory/disk/network plus per-container stats (it mounts `/var/run/docker.sock:ro`). The hub and local agent communicate over a shared unix socket (`./beszel_socket/beszel.sock`, `LISTEN`/`HUB_URL`) rather than an exposed port; additional hosts are onboarded by deploying just the agent elsewhere with the per-system `TOKEN`/`KEY` the hub emits from its "Add System" UI (values kept in `.env` from the vault as `BESZEL_AGENT_TOKEN`/`BESZEL_AGENT_KEY`, never committed). Data persists in `./beszel_data` (PocketBase-backed). Since **home-07 (2026-07-22)** the **NAS** agent (`/volume1/docker/beszel-agent/compose.yaml`, mirrored at `foss-setup/configs/nas/beszel/`) bind-mounts each Synology volume's marker dir at `/extra-filesystems/volumeN:ro` so `/volume1,2,3` break out per-mount in the dashboard — the Docker agent auto-detects anything under `/extra-filesystems` (the `EXTRA_FILESYSTEMS` env var is the systemd/binary form using device names and does NOT apply to the container). volume1 IS the agent's root disk (shown as the main disk gauge); volume2 and volume3 appear as extra filesystems labelled by their `cachedev` device name (`dm-7` and `dm-6` respectively), a Synology device-mapper quirk. The split is guarded by the `nas-beszel-volume-split` check.

## Troubleshooting

- **NAS dashboard shows extra filesystems named `dm-6`/`dm-7` instead of `volume2`/`volume3`.** — Expected. beszel labels an extra filesystem by the block device it resolves for I/O stats; Synology volumes are device-mapper `cachedev` devices, so `/extra-filesystems/volume3` -> `dm-6` and `/extra-filesystems/volume2` -> `dm-7`. volume1 is deduplicated into the agent's root disk (same filesystem) and shows as the main disk gauge. No action needed — usage/percentages are correct; only the label differs.
- **Agent logs show `WebSocket connection failed err="...connection reset by peer"` on startup, then no data appears for the local host.** — This is a benign race: the agent boots before the hub's socket/WS listener is ready and retries — it reconnects seconds later (look for the following `INFO WebSocket connected host=127.0.0.1:8090`). No action needed unless the connected line never appears; then `ssh mini 'cd /opt/stacks/beszel && docker compose restart beszel-agent'`.
- **A remote host (nas/rig) shows offline in the hub, or a newly added system never connects.** — Each system needs its own `TOKEN`/`KEY` pair generated by the hub's Add System dialog — reusing mini's values fails. On the remote host deploy only the agent with `HUB_URL` pointing at the hub and the fresh `KEY`, then confirm the agent log shows `WebSocket connected`.
- **Dashboard unreachable at https://status.tabaska.us even though the container is Up.** — The hub only binds `8090` locally; public access is via Caddy on the `edge` network. Verify the container: `ssh mini 'cd /opt/stacks/beszel && docker compose ps'`, then check Caddy is routing status.tabaska.us to `beszel:8090`. `BESZEL_APP_URL` must be set to the public https URL so links/notifications resolve correctly.
