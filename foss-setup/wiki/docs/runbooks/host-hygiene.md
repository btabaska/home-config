# Host hygiene (mini) — dead services, silent crons, spent timers

Outcome of **fix-39** (quality-gate 2026-07-16 findings M1, M2, M3, M62,
resolved 2026-07-19). The theme: things on the mini that failed or died
**silently** — a dead panel still burning RAM, dropped `/etc` history commits,
a cron exec'ing a directory every midnight for years, a one-shot timer that
failed once and never rescheduled. The checks in
`verification/checks.d/host-hygiene.yaml` watch the silent-failure surface
itself so the *class* cannot recur unnoticed.

## What was retired / fixed

### Pterodactyl LEMP stack (M1)

Game hosting moved to AMP on the rig in 2025; the panel's whole substrate kept
running headless until 2026-07-19: nginx (zero sites), php8.3-fpm, mariadb
(only the `panel` DB), redis, a per-minute root cron
(`artisan schedule:run`), and daily telemetry-failure logs.

- **Removed:** all four services + packages (apt purged), the root cron entry,
  `pteroq.service`, `/var/www/pterodactyl`, `/var/lib/mysql`, `/var/lib/redis`,
  `/usr/local/bin/composer`.
- **Archive (keep until you'd truly never rebuild it):** mini
  `/var/backups/pterodactyl-retired-20260719/` — `panel-db.sql.gz` (verified
  MariaDB dump), `pterodactyl-www.tar.gz`, `pteroq.service`, the old root
  crontab, and the retired `tv_maintenance.sh` (see M62 below). Root-only, 600.
- **Check:** `ptero-lemp-retired` — red if the packages, cron entry, or app dir
  ever reappear.

### etckeeper index.lock races (M2)

`etc-watch.path` commits, the daily autocommit, and **apt's own DPkg etckeeper
hook** raced on `/etc/.git/index.lock`; each loss silently dropped that `/etc`
change from version history (26 failures Jul 01–15).

- **Fix:** every systemd-side invocation now goes through
  `/usr/local/sbin/etckeeper-serialized.sh` — a host-wide `flock` plus a
  retry-with-backoff loop for lock holders we cannot wrap (apt's hook invokes
  `etckeeper` directly). Generated by `scripts/inventory/etckeeper-setup.sh`
  (rerunnable, idempotent) so a rebuilt host gets the same behavior.
- **Proven:** a deterministic test held `index.lock` while a commit fired — the
  unit retried and committed once the lock released, where it previously died
  with exit 128.
- **Check:** `etckeeper-lock-races` — red if the wrapper ever exhausts its
  retries (exit 75) or logs an index.lock fatal.

### Spent one-shot timer (M3)

`media-window-maint.timer` (one-shot `OnCalendar=2026-07-11`,
`Persistent=false`) ran once, **failed** (unpackerr health never came up), and
then sat enabled with `NEXT=n/a` — the failure never rescheduled and nothing
alerted. The underlying objective is since met (unpackerr healthy, queue clean,
scraped by Uptime-Kuma), so the units were deleted, not replaced.

- Also stopped: `apply-static-ip.timer`, the same spent pattern (its unit files
  remain; full removal is fix-43 scope).
- **Check:** `spent-enabled-timers` — flags any timer with `NEXT=n/a` that has
  fired before but is still enabled/active. Event-armed dormant stock timers
  (`LAST=n/a`) pass.

### tv-torrent cleanup cron (M62)

`0 0 * * * /home/btabaska/bin` exec'd a **directory** — it failed every
midnight since Feb 2024 with output going to nonexistent local mail. The
cleanup it named never ran; 27 GB of already-imported torrent leftovers
accumulated.

**Topology note (differs from the audit's assumption):** `/mnt/share/torrents`
is a **local directory on the mini's root LV** — the NAS mount it once was is
gone from fstab, and nothing has written to it since Oct 2025 (the live
seedbox→NAS pipeline runs entirely NAS-side). So pruning here cannot race an
rclone re-copy.

- **Replacement:** `tv-torrent-cleanup.timer` → weekly, Sun 08:30 UTC (4:30 AM
  ET, inside the maintenance window), `Persistent=true`. The service prunes
  top-level entries older than **30 days**, logs every removal to the journal,
  alerts via `ntfy-notify@` on failure, and pings a healthchecks dead-man
  (vault: `healthchecks.tv_cleanup_mini_ping_url`, grace 6 h) on success.
  Config owned by `configs/host/mini/tv-cleanup/`.
- First run 2026-07-19 pruned the whole 27 GB backlog (28 entries, oldest
  Mar 2025); the share is empty.
- **Checks:** `cron-targets-exist` (class: every absolute-path cron command in
  user crontabs + `/etc/cron.d` must be an existing executable file),
  `tv-share-no-ancient-leftovers` (consumer end: nothing older than 40 days may
  exist), `tv-cleanup-timer-armed`.

## If a check goes red

| Check | Likely cause | Response |
|---|---|---|
| `ptero-lemp-retired` | Something reinstalled a LEMP package or recreated the cron | Find what installed it (`/var/log/apt/history.log`, `/etc` git log); nothing on the mini should need host nginx/mariadb/redis — containers bring their own |
| `etckeeper-lock-races` | A long-lived etckeeper/git process is holding `index.lock` > ~2 min | `sudo fuser -v /etc/.git/index.lock`; if stale with no holder, remove the lock file, then `sudo systemctl start etckeeper-commit.service` |
| `spent-enabled-timers` | A one-shot timer fired its last elapse and stayed armed | If the job still matters, give it a recurring `OnCalendar` or `Persistent=true`; if not, `systemctl disable --now` and delete the unit files |
| `cron-targets-exist` | A crontab entry's target was deleted/moved | Fix the path or remove the entry — this is exactly how M62 rotted for 2.5 years |
| `tv-share-no-ancient-leftovers` | Cleanup timer dead, or it runs but deletes nothing | `systemctl list-timers tv-torrent-cleanup.timer`; `journalctl -u tv-torrent-cleanup`; run `sudo systemctl start tv-torrent-cleanup.service` and watch the journal |
| `tv-cleanup-timer-armed` | Timer disabled or unit files missing | Reinstall from `configs/host/mini/tv-cleanup/`, `systemctl enable --now tv-torrent-cleanup.timer` |

The healthchecks dead-man (`tv-torrent-cleanup (mini)`) alerts independently if
the weekly ping stops — the local checks catch a disarmed timer *before* the
grace window does.
